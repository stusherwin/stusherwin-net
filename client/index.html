Technical details

Written in: Elm Haskell PostgreSQL

I'm really into strongly-typed functional languages. I fell in love with Haskell when I first disovered Learn You a Haskell For Great Good and I've been coding in it ever since (check out some of my other projects). I love Haskell for how it separates pure functions from code with side-effects using monads, which really forces you to push I/O to the boundaries of your application.

Languages that aren't as strict allow you to get away with code that has hidden effects, which leads to unpredictability and creates complexity in managing that unpredictability. Haskell forces you to deal with that complexity up front, which means a bit more work to get your code to build, but you gain so much more by being able to make sweeping changes to your codebase without introducing hard-to-fix bugs, and the end result is much more stable applications with very few runtime errors.

Once I learned how to do this with my server-side code, I wasn't happy with not being able to do the same for my front-end code. I found React and TypeScript to be a good working solution but when I discovered Elm I knew I had found what I was looking for. It's not perfect and it's (intentionally) quite limited in its type system, it doesn't have a concept of monads which would make things much less verbose, but I can put up with this for the joy of writing in pure functions.

Technical details

Written in: React TypeScript Haskell PostgreSQL

Before I discovered Elm I loved React for it's functional data flow - components are rendered from state, and actions update the state, not the components. This eliminates a lot of the complexity from state management, and with TypeScript providing type-safety it's a pretty powerful combination.

However, I don't think it goes quite far enough - Elm also has the type safety but it pushes the functional data flow to its logical conclusion - everything is a pure function!

Configit

Configit make software to solve configuration problems. For example, when buying a car you have a set of options, and selecting any one of the options opens up new options and closes off others. For a large number of options and rules this becomes unmanageable very quickly, and you need to be able to know if the rules are consistent and you can always choose a valid set of options. Their technology can verify this analytically.

It's also another Agile shop, with emphasis on TDD and some Extreme Programming ideas like pair programming. We quite often paired on complex tasks which was really challenging and tiring but also eye-opening.

I worked on various parts of their main web app for Jaguar Land Rover that uses Configit's technology to manage the conceptual design process for their new cars. I spent a big chunk of this time developing a task runner system to distribute intensive tasks across a network of computers.

Working here gave me a lot of insight into when TDD works well and when it doesn't, e.g. having a comprehensive set of tests around the boundaries of an application (not the internals) makes a huge difference to your confidence in making changes. But there's also times when it will hold you back, e.g. if the app is at a very experimental stage. There's a time and place for every technique but the key is not to be religious about anything.
